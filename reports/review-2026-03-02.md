**Scope:** How we track performance data (agent hours, invocations); cross-reference with initiatives. Read-only review; no file edits.

## Why “agent hours” could go down

**Short answer:** We don’t have a separate system log or no-SQL DB for agent invocations. The only source is the **Dolt `event` table**. The number you see (“118 yesterday”) is the sum of all **started → done** elapsed time (in hours) across every task that has a `done` event, computed live from that table.

So “agent hours” can **decrease** only if:

1. **Different Dolt data** — You’re on a different repo, a different `.taskgraph` (e.g. fresh clone, another machine), or you’ve checked out an older Dolt commit/branch that has fewer events.
2. **Direct DB changes** — Someone ran destructive SQL (against our rules) and deleted event rows; the app also has a `no_delete_event` trigger to block that.

Under normal use (same repo, same branch, no deletes), the value is **monotonic**: it only goes up as more tasks are marked done. So if you saw 118 and then a lower number, the most likely explanation is a **different database or branch**, not a second “system log” being corrected.

## How we track performance data (single source of truth)

| What                                                  | Source                       | Where                                                                                                                      |
| ----------------------------------------------------- | ---------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| **Total Agent Invocations**                           | Dolt `event` table           | `COUNT(*)` where `kind = 'done'` in `src/cli/status.ts`                                                                    |
| **Total Agent hours**                                 | Dolt `event` table           | Sum of (done time − latest started time per task), in seconds, ÷ 3600; rounded. Same file, single `agentMetricsSql` query. |
| **Investigator runs / fix rate**                      | Same table                   | Done events with `body.agent = 'investigator'`; fix rate from evidence.                                                    |
| **Per-task elapsed, plan duration, velocity, tokens** | Same table + `tg done` flags | `tg stats`, `tg stats --plan`, `--timeline`, `--recovery` in `src/cli/stats.ts`                                            |

There is **no** separate “system log” or no-SQL DB for invocations. The **only** implemented store for these metrics is **Dolt** (the versioned SQL DB in `.taskgraph/dolt`).

**Optional telemetry (designed, not implemented):** The plan **Agent Context Sync** (26-03-01) and `docs/agent-context.md` describe a **SQLite** store (`.taskgraph/agent_context.db`) that would collect `[tg:event]` lines from terminal files and expose a query API. The CLI has `tg agent-context collect` / `query` / `status` and spawns:

- `scripts/collect-agent-events.ts`
- `scripts/query-agent-events.ts`

Those **scripts do not exist** in the repo. So:

- There is **no** second “no-SQL” (or SQLite) log of agent invocations that we actually built.
- “Agent hours” and “Total Agent Invocations” are **not** coming from that; they come only from Dolt.
- If you run `tg agent-context collect` or `query`, they will fail (scripts missing).

---

## Code paths (concise)

- **Reads:** Status dashboard and vanity line use one `agentMetricsSql` in `status.ts` (no `--plan` filter; always global). Stats use multiple queries in `stats.ts` (plan, timeline, recovery, token usage from done body). MCP, show, context, health-check and utils read `event` for history and started/done.
- **Writes:** Only `tg start` (inserts `started`) and `tg done` (inserts `done`) change agent hours and invocations. Other commands (note, cancel, block, gate, import, split, etc.) don’t affect those two metrics.
- **Cache:** `status-cache.ts` caches the result of `fetchStatusData` (including the agent-metrics query) for latency; it does not define or store a separate performance store.

## Summary and next steps

- **Overall:** Performance data is tracked in **one place**: the Dolt `event` table. “Agent hours” and “Total Agent Invocations” are derived from it in `status.ts` and cannot go down unless you switch to a different Dolt repo/branch or events are deleted (forbidden). There is no separate system log or no-SQL DB for invocations; the SQLite agent_context was designed but never implemented (scripts missing).

**Top follow-ups:**

1. **Implement or narrow Agent Context** — Either add `scripts/collect-agent-events.ts` and `scripts/query-agent-events.ts` per the plan and docs, or document that metrics are Dolt-only and have `tg agent-context` fail with a clear “not implemented” message.
2. **Clarify in docs** — In `docs/performance.md` (or a short “Data sources” subsection), state explicitly that the single source of truth for agent invocations and agent hours is the Dolt `event` table, and that agent_context (when/if implemented) is optional telemetry only and does not drive those numbers.
3. **Optional: plan-scoped metrics** — If you want “agent hours for this plan only”, add an optional `--plan`-scoped view in status or stats, with a clear label so it’s distinct from the global “Total Agent hours”.
4. **Fix status-live --json flakiness** — Address stdout flush vs `process.exit(0)` so the 3 tests are stable in gate:full (see memory/docs).

# Task-Graph documentation assessment report

## 1. System narrative

**Where the story is clear**

- **README.md** gives a short "how it works": Quick Start (lines 62–92) is a step-by-step flow (init → import → next → start → done → status) with concrete commands. The "What This Is" / "What This Isn't" (lines 13–25) set scope. The Architecture section (lines 95–149) has a **Mermaid flowchart** (CLI → Dolt, import/exec/export/MCP) and a **layers table** (cli, db, domain, plan-import, export), with a pointer to `docs/architecture.md`.
- **docs/README.md** frames the system in one paragraph ("centaur development", "issue graph (tasks + dependencies + events) backed by Dolt") and points to Glossary, Architecture, Schema, and a domain index.
- **docs/architecture.md** continues the narrative: Data Store (Dolt, `dolt sql`, versioning), Execution Surface (CLI), repo layout, config, then Data Flow and Error Handling with a **Mermaid diagram** (layers + branch-per-agent + error flow).

**Where it's missing or thin**

- There is no **single "system story" doc** that walks: entrypoints (CLI, MCP) → data (Dolt, project/task/edge/event) → main flows (plan→import→project; next→start→work→done; skill→lead→workers; worktree/merge) → outcomes (done tasks, exported markdown, gate). That story is split across README, architecture, agent-contract, and multi-agent.
- **Plan → graph → execution** is implied (README Quick Start, agent-contract "Agent Operating Loop") but not one place that says: "Plan file → import → project + tasks in Dolt → next returns runnable tasks → start claims → work → done writes events and updates status."
- **MCP** is listed as an entrypoint in README and architecture but has no narrative role in the main "how the system works" flow; it's "read task/plan data" without tying it into the same data model and flows.

**Summary:** The system narrative exists in pieces (README + architecture + agent-contract). A single "System overview" or "How the system works" section (in README or a short doc) that strings entrypoints → data → flows → outcomes would close the gap.

## 2. Key domains

**What's in place**

- **docs/domains.md** is the **map**: table of domain slug → doc → one-line description. It covers architecture, schema, cli-reference, cli, cli-tables, plan-import, plan-format, error-handling, testing, infra, agent-contract, agent-strategy, multi-agent, mcp, glossary, plus backend stub, recommended-packages, performance, agent-context.
- **docs/architecture.md** defines the **technical bounded contexts** in "Repository Layout" and "Data Flow": `cli/`, `db/`, `domain/`, `export/`, `plan-import/` with clear roles. The layer diagram ties them together.
- **docs/README.md** groups docs into Core, CLI, Agent, Development, Planning and links to domains.md as "domain slug index."

**Gaps and boundary issues**

- **Export** is a layer in architecture and README but has **no dedicated domain doc** (no `docs/export.md`). Export behavior appears in `cli-reference.md` (export mermaid/dot/markdown). So "export" is a code layer but not a documented bounded context.
- **Agent system** is split across **agent-contract**, **agent-strategy**, **multi-agent**, **docs/leads/**, and **.cursor/agents/**. domains.md lists agent-contract, agent-strategy, multi-agent but not "leads" or "skills" as first-class domains; the lead registry lives in `docs/leads/README.md`. So "agent system" is a logical domain without one clear doc that states "this doc owns: orchestrator, skills, leads, workers, dispatch, escalation."
- **Plan vs project** is well defined in **glossary** and schema; plan-import owns "file → graph." The boundary between **plan-format** (file shape) and **plan-import** (parsing and DB upsert) is clear in the doc list; a short cross-link in both ("for DB side see plan-import" / "for file shape see plan-format") would make the map easier to follow.
- **domains.md** does not list **AGENT.md** or **.cursor/rules**; the docs-sync rule says AGENT.md is canonical for the agent contract. So the "agent" domain spans repo root and .cursor; the index doesn't say "for contract read AGENT.md + agent-contract.md."

**Summary:** domains.md + architecture give a navigable map for code-oriented domains (cli, db, domain, plan-import, schema). The main gaps are: export as a documented domain, a single "agent system" entry that points to AGENT.md + agent-contract + agent-strategy + multi-agent + leads, and explicit cross-links between plan-format and plan-import.

## 3. Flows and patterns of use

**Documented flows**

- **Plan → import → next → start → work → done:**  
  README Quick Start; agent-contract "Agent Operating Loop" (select next → show → start → execute → done) and "Plan Creation and Review" (create plan → pause → interpret "proceed" / "add only" / "thanks"). No single end-to-end diagram.
- **Skill → lead → workers:**  
  agent-strategy (orchestrator, leads, workers, "Decision Tree"); docs/leads/README.md (skill → lead → workers → synthesis); README "Agent system" and skills/lead tables. The **sequence** (user says "work" → /work skill → execution lead → implementer + reviewer) is described in prose and tables, not as a step-by-step flow diagram.
- **Worktree / per-plan branch:**  
  multi-agent.md "Worktrunk — sub-agent worktree backend" (per-plan branch and worktree, merge target plan branch, parallel tasks, backend selection). Steps for `tg start --worktree` and `tg done --merge` are numbered. **Merge flow** (task branch → plan branch → later main) is text-only; no diagram.
- **Blocking and escalation:**  
  agent-contract "When Blocked," "Escalation ladder," "gate:full Orchestration Rules," and "Stuck agent / watchdog." subagent-dispatch.mdc is referenced for the escalation tree; no flowchart in the docs themselves.
- **Plan creation and review:**  
  agent-contract table: "proceed" → import + execute; "just add the tasks" → import only; "thanks" → no action.

**Where step-by-step or diagrams would help**

- **Execution loop (single agent):** One page or diagram: "tg next (runnable) → pick task → tg show → tg start [--agent] [--worktree] → work in worktree → tg done [--merge] --evidence" with a note that "tg done must run from repo root" (per taskgraph-workflow) and where gate:full is run (plan worktree, not main). Would tie README Quick Start to agent-contract and multi-agent.
- **Multi-agent worktree flow:** Diagram: plan with hash_id → first `tg start --worktree` creates plan branch + plan worktree → each task gets task worktree from plan branch → `tg done --merge` merges task into plan branch → plan worktree accumulates; optional "plan complete" step merging plan branch to main. Would make multi-agent.md's "Merge target is the plan branch" and "Parallel task handling" easier to follow.
- **Skill → lead → worker dispatch:** Mermaid or numbered flow: User input → skill trigger → lead created → lead dispatches worker(s) → worker runs (e.g. implementer/reviewer) → result back to lead → synthesis → orchestrator. docs/leads/README.md "Standard skill anatomy" asks for a "Decision tree (mermaid flowchart)" per skill; a **generic** skill→lead→worker flowchart in agent-strategy or leads/README would anchor all skills.
- **Task/project state machine:** Task status (todo → doing → done; todo/doing → blocked; canceled) and project status (draft ↔ active) are in schema and glossary but not drawn. A small state diagram would clarify valid transitions and when project becomes active.

**Summary:** Main flows (plan→import→execution, skill→lead→workers, worktree/merge) are documented in prose and tables. Adding one execution-loop narrative (with optional diagram), one worktree/merge diagram, one skill→lead→worker diagram, and a task/project state diagram would make patterns of use much easier to follow.

Highest value for new Mermaid: (1) worktree/merge flow, (2) task/project state machine, (3) agent dispatch (skill→lead→workers), (4) execution loop, (5) data flow plan→Dolt→CLI.

## 5. Gaps and recommendations

**Concrete gaps**

- **No single "system narrative" page** that walks entrypoints → data → flows → outcomes.
- **Export** is a code layer but has no domain doc (only cli-reference and architecture layer table).
- **Agent system** is spread across AGENT.md, agent-contract, agent-strategy, multi-agent, leads; no one doc that states "this is the agent domain and here's the index."
- **Execution loop** is in README and agent-contract but not one place with worktree and gate:full rules (e.g. "tg done from repo root," "gate:full from plan worktree").
- **Worktree/merge flow** is only prose; no diagram (plan branch, task branches, merge order).
- **Task and project state transitions** are only in schema/glossary text; no state diagram.
- **Skill → lead → worker** has no shared Mermaid; only per-skill decision trees are required.
- **domains.md** doesn't reference AGENT.md or .cursor/rules; export has no row.

**3–5 high-impact recommendations**

1. **Add a short "How the system works" narrative** (in README or `docs/overview.md`): one page that walks (1) entrypoints (CLI, MCP), (2) data (Dolt, project/task/edge/event), (3) main flows (plan→import→project; next→start→work→done; skill→lead→workers; worktrees and merge), (4) outcomes (done tasks, export, gate). Link to architecture, schema, agent-contract, multi-agent. This gives a single "system story" for humans and agents.

2. **Add two Mermaid diagrams:**
   - **multi-agent.md:** Worktree/merge flow (main, plan branch, task branches; start --worktree; done --merge; plan→main when complete).
   - **schema.md or glossary.md:** Task and project state machine (states and allowed transitions).  
     These answer "where does my work go?" and "what states can a task/project be in?"

3. **Document the execution loop in one place** (e.g. agent-contract or README): one subsection that lists steps (next → show → start [--agent] [--worktree] → work → done [--merge] --evidence), with rules: "tg done from repo root," "gate:full from plan worktree after tasks merged," "WORKTREE_PATH for implementer." Optionally add a small Mermaid "Execution loop" diagram.

4. **Tighten the agent-domain map:** In **docs/README.md** (or a short "Agent system" subsection in agent-strategy): state that the agent system is documented in AGENT.md (canonical contract), agent-contract.md (expanded contract), agent-strategy.md (orchestrator/leads/workers), multi-agent.md (coordination/worktrees), and docs/leads/ (lead registry and per-lead docs). Add to **domains.md** a note or row that "agent" contract and workflow live in AGENT.md + agent-contract + agent-strategy + multi-agent + docs/leads/.

5. **Add a generic "Skill → lead → worker" Mermaid** in **docs/agent-strategy.md** or **docs/leads/README.md**: one flowchart (user → skill → lead → dispatch worker(s) → synthesis → orchestrator). Keep per-skill decision trees in skills; use this as the shared pattern so every skill doesn't have to redraw the same structure.
